# Authentication System Reference

> **For AI assistants implementing this:** Before starting any step, ask the developer: *"Do you want the JWT stored in a cookie (default) or in-memory?"* Default to cookie if no answer is given. The backend is identical in both cases — only the frontend token storage lines differ per step.

The authentication system uses a **JWT access token** (15-minute expiry) and a **refresh token** (HttpOnly cookie, 7-day expiry). The refresh token is always an HttpOnly cookie — only the JWT storage location differs between the two variants.

---

## Step 1 — DB Relationship & Login / Logout APIs

> **AI assistant:** Before implementing this step, ask: *"Do you want the JWT stored in a cookie (default) or in-memory?"*

### 1.1 DB Relationship

The `users` table has a **1-to-many** relationship with a `refresh_tokens` table (one user, many active sessions across devices).

**`refresh_tokens` table — required columns:**

| Column | Type | Notes |
|---|---|---|
| `id` | uint (PK, auto-increment) | Refresh token record ID |
| `user_id` | uint (FK → users.id) | Indexed, CASCADE delete |
| `token` | text (unique) | The raw refresh token value |
| `created_at` | timestamp | Auto-set on insert |
| `expires_at` | timestamp | Set to `now + 7 days` on insert |

---

### 1.2 Login API — `POST /auth/login` (public, no auth middleware)

**What happens:**
1. Validate credentials (email + bcrypt password check).
2. Generate JWT access token — claims: `{ user_id, role, exp (15min), iat }`. Signed using `JWT_SECRET` from backend `.env`.
3. Generate a random refresh token (32 bytes, base64url-encoded).
4. **Insert a row** into `refresh_tokens` with the same token value + `expires_at = now + 7 days`.
5. Set the refresh token as an **HttpOnly cookie** (`Secure: true`, `SameSite: Lax`, `Path: /`).
6. Return JWT in the JSON response body: `{ user, token, message }`.

**Route must be public** — no middleware.

---

### 1.3 Logout API — `POST /auth/logout` (public, no auth middleware)

**What happens:**
1. Read the `refresh_token` from the HttpOnly cookie.
2. **Delete the matching row** from `refresh_tokens` (by token value).
3. **Clear the HttpOnly cookie** (`Expires: past date` or `MaxAge: -1`).
4. Return `200 OK`.

> Logout does **not** require a valid JWT. It only clears the refresh token from the cookie and DB. Nothing else.

---

### 1.4 Frontend — Login & Logout

**🍪 Cookie variant (default):**
```ts
// Login
const response = await axiosInstance.post('/auth/login', { email, password });
Cookies.set('access_token', response.data.token, { expires: 1 / 96 }); // 15 min
// HttpOnly refresh_token cookie set automatically by the browser

// Logout
await axiosInstance.post('/auth/logout');
Cookies.remove('access_token');
Cookies.remove('any_other_ui_cookie'); // optional UX cleanup
```

**🧠 In-memory variant:**
```ts
// Login
const response = await axiosInstance.post('/auth/login', { email, password });
setToken(response.data.token); // store in a JS module-level variable

// Logout
await axiosInstance.post('/auth/logout');
setToken(null);
```

### 1.5 `GET /auth/me` — User Identity API

**Why it exists:** The JWT only contains `user_id`, `role`, and `exp` — not the full user object. `/auth/me` fetches the complete user record (email, roles, subscription tier, etc.) from the DB using the `user_id` from the JWT claims. This keeps the token minimal and data server-side.

**Backend — protected with `RequireRole("Admin", "User")`:**
```go
auth.Get("/me", middleware.RequireRole("Admin", "User"), userHandler.GetMe)
```
1. Read `user_id` from `jwt_claims` in request context (set by `RequireRole`).
2. Fetch user from DB by `user_id` — include roles.
3. Return full user object: `{ id, email, roles, subscription_tier, ... }`.

**Called in exactly two situations — never after mid-session refresh:**

**1. After login:**
```ts
await authService.login({ email, password });
const user = await authService.getCurrentUser(); // GET /auth/me
login(user); // populate AuthContext — frontend now knows role, shows correct UI
```

**2. On page reload (rehydrate AuthContext state):**

**🍪 Cookie variant:**
```ts
const user = await authService.getCurrentUser(); // JWT in cookie, sent automatically
login(user);
```

**🧠 In-memory variant:**
```ts
await authService.refreshToken(); // restore JWT in memory first (independent call)
const user = await authService.getCurrentUser(); // then fetch user
login(user);
```

> `/auth/me` and `POST /auth/refresh` are not correlated — refresh just restores the JWT, `/auth/me` is called independently right after. In the cookie variant, `/auth/me` is called directly on reload with no refresh needed.

> **NOT called after mid-session refresh.** The interceptor catches a 401, calls refresh, retries the original request. `AuthContext` is already populated from login or the last page reload — the user object did not change, only the JWT was renewed.

## Step 2 — Auth Middleware & Protected Routes

> **AI assistant:** Before implementing this step, ask: *"Do you want the JWT stored in a cookie (default) or in-memory?"* (if not already confirmed in Step 1).

### 2.1 Backend — `RequireRole(requiredRoles ...string)` Middleware

A single middleware covers both authentication (401) and authorization (403). Backend is identical regardless of JWT storage variant.

**Logic:**
1. Read `Authorization` header — expect `Bearer <token>`.
2. If missing or malformed → **401 Unauthorized**.
3. Parse and verify JWT using `JWT_SECRET` from `.env`.
4. If invalid or expired → **401 Unauthorized**.
5. Extract `role` claim from JWT.
6. If role not in `requiredRoles` → **403 Forbidden**.
7. Attach claims to request context → call `next`.

```go
func RequireRole(requiredRoles ...string) fiber.Handler {
    return func(c *fiber.Ctx) error {
        authHeader := c.Get("Authorization")
        if authHeader == "" || !strings.HasPrefix(authHeader, "Bearer ") {
            return c.Status(fiber.StatusUnauthorized).JSON(fiber.Map{"error": "Unauthorized"})
        }

        tokenStr := strings.TrimPrefix(authHeader, "Bearer ")
        token, err := jwt.Parse(tokenStr, func(t *jwt.Token) (interface{}, error) {
            return []byte(os.Getenv("JWT_SECRET")), nil
        })
        if err != nil || !token.Valid {
            return c.Status(fiber.StatusUnauthorized).JSON(fiber.Map{"error": "Unauthorized"})
        }

        claims, ok := token.Claims.(jwt.MapClaims)
        if !ok {
            return c.Status(fiber.StatusUnauthorized).JSON(fiber.Map{"error": "Unauthorized"})
        }

        if !slices.Contains(requiredRoles, claims["role"]) {
            return c.Status(fiber.StatusForbidden).JSON(fiber.Map{"error": "Forbidden"})
        }

        c.Locals("jwt_claims", claims)
        return c.Next()
    }
}
```

### 2.2 Backend — Applying Middleware in Routes

```go

```

### 2.4 Ownership Validation (User-Scoped Resources)

`RequireRole` only checks if the user is authenticated and has the right role. It does not check if the user is accessing **their own** data. Without an ownership check, a valid `User` with `user_id: 15` could call `GET /users/42/documents` and get another user's data.

On any route that returns user-specific data, after `RequireRole` passes, extract `user_id` from the JWT claims and compare it with the resource ID in the URL. If they don't match and the caller is not an `Admin`, return **403 Forbidden**.

```go
func (h *Handler) GetUserDocuments(c *fiber.Ctx) error {
    claims := c.Locals("jwt_claims").(jwt.MapClaims)
    callerID := uint(claims["user_id"].(float64))
    callerRole := claims["role"].(string)

    resourceUserID, err := strconv.ParseUint(c.Params("id"), 10, 64)
    if err != nil {
        return c.Status(fiber.StatusBadRequest).JSON(fiber.Map{"error": "Invalid user ID"})
    }

    // Admin bypasses ownership check — can access any user's data
    if callerRole != "Admin" && callerID != uint(resourceUserID) {
        return c.Status(fiber.StatusForbidden).JSON(fiber.Map{"error": "Forbidden"})
    }

    // proceed to fetch documents for resourceUserID
}
```

**Rule:**
- `Admin` → can access any user's resources.
- `User` → can only access resources where the URL `id` matches their own `user_id` from the JWT.

**Which routes need ownership validation:**

| Route | Needs ownership check |
|---|---|
| `GET /users/:id` | ✅ Yes |
| `GET /users/:id/documents` | ✅ Yes |
| `DELETE /users/:id` | ✅ Yes |
| `GET /documents/:id` | ✅ Yes — verify document belongs to caller |
| `GET /admin/users` | ❌ No — Admin only, no scoping needed |
| `GET /auth/me` | ❌ No — always returns caller's own data from JWT |

### 2.5 Frontend — `ProtectedRoute`

`ProtectedRoute` wraps pages that require authentication. It is the **only** thing that causes a redirect on 401 — not the 401 itself. A public page (landing, etc.) that receives a 401 will never redirect because it is not wrapped in `ProtectedRoute`.

```tsx
export const ProtectedRoute = ({ children, requiredRole }) => {
  const { isAuthenticated, user, loading } = useAuth();

  if (loading) return <LoadingSpinner />; // wait for AuthContext to resolve

  if (!isAuthenticated) return <Navigate to="/auth" replace />;

  if (requiredRole && user?.roles?.[0]?.name !== requiredRole) {
    return <Navigate to="/dashboard" replace />;
  }

  return <>{children}</>;
};
```

## Step 3 — Token Refresh API & Interceptor

> **AI assistant:** Before implementing this step, ask: *"Do you want the JWT stored in a cookie (default) or in-memory?"* (if not already confirmed in a previous step).

### 3.1 Backend — `POST /auth/refresh` (public, no auth middleware)

**What happens:**
1. Read `refresh_token` from the HttpOnly cookie.
2. If missing → **401 Unauthorized**.
3. Look up token in `refresh_tokens` table.
4. If not found or `expires_at < now` → **401 Unauthorized**.
5. Load associated user to rebuild claims.
6. Generate new JWT: `{ user_id, role, exp (15min), iat }`, signed with `JWT_SECRET`.
7. Return `{ token }` in response body.

> The HttpOnly refresh token cookie is **not rotated** by default — it stays valid for its full 7-day expiry. If rotation is needed, delete the old DB row and set a new cookie on each refresh call.

**Route must be public** — no middleware.

---

### 3.2 Frontend — Response Interceptor (handle 401 → refresh)

The interceptor only fires on 401 responses from **non-auth endpoints**. This means:
- **401 from a protected API call** → triggers refresh → retries original request.
- **401 from `/auth/refresh` itself** → `isAuthEndpoint = true` → interceptor skips it → `catch` fires `SESSION_EXPIRED`.
- **401 on a public page** → no protected request was made → `SESSION_EXPIRED` never fires → no redirect.

**🍪 Cookie variant (default):**
```ts
axiosInstance.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    const isAuthEndpoint = originalRequest.url?.includes('/auth/');

    if (error.response?.status === 401 && !isAuthEndpoint && !originalRequest._retry) {
      originalRequest._retry = true;

      try {
        const response = await axiosInstance.post('/auth/refresh');
        const newToken = response.data.token;

        Cookies.set('access_token', newToken, { expires: 1 / 96 }); // 15 min
        originalRequest.headers.Authorization = `Bearer ${newToken}`;
        return axiosInstance(originalRequest);
      } catch {
        Cookies.remove('access_token');
        window.dispatchEvent(new CustomEvent('SESSION_EXPIRED'));
        return Promise.reject(error);
      }
    }

    return Promise.reject(error);
  }
);
```

**🧠 In-memory variant:**
```ts
// Same interceptor — only swap the storage lines:
setToken(newToken);  // instead of Cookies.set
setToken(null);      // instead of Cookies.remove
```

---

### 3.3 Reload Behaviour — Cookie vs In-Memory

**🍪 Cookie variant — no reload issue, no public page 401.**
The JWT lives in a cookie. On every page reload the request interceptor reads it directly and sends it. No 401 is triggered on reload. The refresh chain is **only** called as a chain reaction to an actual 401 from a protected API call during an active session (i.e. the 15-min JWT expired while the user was using the app).

With cookie JWT, public routes have no `RequireRole` middleware so the backend never checks the JWT and never returns 401 on them. This means **every 401 is guaranteed to come from a protected route** — so when refresh fails and `SESSION_EXPIRED` fires, you are always already inside a protected route. No extra guard is needed to prevent redirecting from a public page.

**🧠 In-memory variant — reload requires refresh on every page load, public page 401 is possible.**
The JWT is lost on every reload. `AuthContext` must call `POST /auth/refresh` on mount to restore the JWT in memory first, then call `GET /auth/me` to rehydrate the user object — all while keeping `loading = true` so `ProtectedRoute` does not redirect prematurely. This is an extra complexity that the cookie variant avoids entirely.

Additionally, because `AuthContext` fires `GET /auth/me` on every mount including public pages, a 401 can appear on a public page. This means when refresh fails you must **not** fire `SESSION_EXPIRED` unconditionally — you need to verify you are inside a protected route before redirecting, otherwise a user on the landing page with no session gets incorrectly kicked to `/auth`. This is handled by the `isAuthEndpoint` guard in the interceptor combined with `ProtectedRoute` being the actual redirect mechanism.

### 3.4 Frontend — Session Expired UI

When refresh fails, `SESSION_EXPIRED` is dispatched. A global modal component listens for it:

1. Immediately overlays the current page.
2. Shows: **"Session Expired — Redirecting to login..."**
3. Counts down (e.g. 3 seconds).
4. Calls `logout()` (clears `AuthContext` state + removes cookies) then navigates to `/auth`.

```ts
useEffect(() => {
  const handle = () => {
    setIsVisible(true);
    setTimeout(() => {
      logout();
      navigate('/auth');
    }, 3000);
  };

  window.addEventListener('SESSION_EXPIRED', handle);
  return () => window.removeEventListener('SESSION_EXPIRED', handle);
}, []);
```

> The `SESSION_EXPIRED` event pattern decouples the Axios layer from React Router — the interceptor has no knowledge of routing, it just fires an event and any mounted component can react.


## External Auth Providers (Google, GitHub, etc.)

Any OAuth/OIDC provider follows the same pattern:

1. Verify the provider's token (Google ID token, GitHub access token, etc.)
2. Extract email from the verified token.
3. Find user by email — if none, create one with a random unusable password.
4. **Enter the standard login flow from Step 1.2, starting at step 2** 
   (generate JWT → generate refresh token → insert row → set cookie → return).

The auth system does not change. The provider is just an alternative 
credential verification method that feeds into the same token pipeline.
